探索 MonoGame 的全新资源管理项目 (ContentBuilder)

No description
 2026.4.22  2026.4.22  奇技与淫巧  2524  6 分钟

新的内容管理 (Content Builder Project) 类似于自己写一个命令行内容管理工具。

使用这个特性要求使用 3.8.5+ 的版本，现在是预览版。

安装这套项目模板的方式如下：

>dotnet new install MonoGame.Templates.CSharp@3.8.*-preview.*

完成后可以通过下面的指令：

# 查看已经安装的橡木模板
>dotnet new list MonoGame
#创建一个 MonoGame-Windows 项目
>dotnet new mgwindowsdx

创建 ContentBuilder

在游戏项目的评级目录，也就是 .csproj 文件所在目录的上一级目录进行操作。

我们将新建一个与游戏项目分离的 ContentBuilder 项目，首先，使用 CMD 或 Powershell 打开这一层的目录：

# 新建文件夹
>mkdir MyContentBuilder
# 新建 ContentBuilder 项目
>dotnet new mgcb -o MyContentBuilder
# 把新建的 ContentBuilder 添加到游戏主项目的解决方案中
>dotnet sln add MyContentBuilder/MyContentBuilder.csproj

现在，顶层的文件夹结构应该是

  • 游戏主项目
  • ContentBuilder 项目
  • 整个项目的解决方案文件 (.sln 或 .slnx)

修改游戏项目

从游戏项目中移除旧的资源管理工具链。

第一步，移除 dotnet tool ，也就是旧的 mgcb-editor 。直接删除 .config 文件夹即可。

第二部移除原来的 Content 文件夹，将资源移动到新建的 ContentBuilder 项目的 Assets 文件夹中。

第三部，移除所有与 MonoGame 内容管理相关的包引用。

打开 .csproj 文件，，向下找到声明 package 引用的 <ItemGroup> 节点。

可能存在多个 <ItemGroup> 节点，一般是随在所有的 <ProjectGroup> 节点后的第一个 <ItemGroup> 块。

内容类似于这样

  <ItemGroup>
    <PackageReference Include="MonoGame.Framework.WindowsDX" Version="3.8.*" />
    <PackageReference Include="MonoGame.Content.Builder.Task" Version="3.8.*" />
    <!-- 其他引用 -->
    <PackageReference Include="Newtonsoft.Json" Version="13.0.3" />
  </ItemGroup>

删掉这个 <PackageReference Include="MonoGame.Content.Builder.Task" Version="3.8.*" /> 。

另外，改一下版本号，匹配 ContentBuilder 项目中引用的包的版本。类似于 <PackageReference Include="MonoGame.Framework.WindowsDX" Version="3.8.5-preview.*" /> 。

在游戏项目中集成 ContentBuilder

在 .csproj 文件的末尾 </Project> 标签前面新起一行，添加如下配置：

  <Target Name="BuildContent" BeforeTargets="BeforeCompile">
    <PropertyGroup>
      <BuilderRoot>$(SolutionDir)ContentBuilder</BuilderRoot>
      <ContentCommand>$([System.IO.Path]::Combine('$(BuilderRoot)','bin\Debug\ContentBuilder.exe'))</ContentCommand>
      <AssetsDir>$([System.IO.Path]::Combine('$(BuilderRoot)', 'Assets'))</AssetsDir>
      <ContentOutput>$(ProjectDir)$(OutputPath)</ContentOutput>
      <ContentTemp>$(ProjectDir)$(IntermediateOutputPath)</ContentTemp>
      <ContentArgs>build -p $(MonoGamePlatform) -s $(AssetsDir) -o $(ContentOutput) -i $(ContentTemp)</ContentArgs>
    </PropertyGroup>
    <MSBuild Projects="$(BuilderRoot)\BBBallContentBuilder.csproj" Targets="Build" RemoveProperties="Configuration;TargetFramework;RuntimeIdentifier;RuntimeIdentifiers"/>
    <Exec Command="$(ContentCommand) $(ContentArgs)" WorkingDirectory="$(MSBuildThisFileDirectory)..\" CustomErrorRegularExpression="\[E\] .+" CustomWarningRegularExpression="\[W\] .+" />
  </Target>

需要注意几个地方。

<AssetsDir>$([System.IO.Path]::Combine('$(BuilderRoot)', 'Assets'))</AssetsDir>

这一行是位于 ContentBuilder 下的 Assets 文件夹，如果你修改了资源文件路径，务必修改这个节点以对齐：

<ContentCommand>$([System.IO.Path]::Combine('$(BuilderRoot)','bin\Debug\ContentBuilder.exe'))</ContentCommand>

记得把这一行指向 ContentBuilder 实际编译出来的可执行文件：

<BuilderRoot>$(SolutionDir)ContentBuilder</BuilderRoot>
<MSBuild Projects="$(BuilderRoot)\ContentBuilder.csproj" Targets="Build" RemoveProperties="Configuration;TargetFramework;RuntimeIdentifier;RuntimeIdentifiers"/>

确保这两个路径与实际创建的 ContentBuilder 项目一致：

最后，不要轻易动用格式化 ，项目文件中添加不必要的换行或空格可能导致编译失败。

现在，在主项目解决方案文件夹下键入

dotnet build

流程将自动构建自定义的 ContentBuilder ，然后调用它处理资源，并将处理结果输出到主游戏项目的输出路径中。比如 GameSolution/Game/bin/Debug/net8.0-windows/Content

Builder.cs

接下来的部分我们对 ContentBuilder 项目进行一些探索，主要是 Builder.cs 这个文件；并展示怎么自定义一个完全可控的资源处理流程，包含将资源导入到处理到写出到读取的四个步骤。

首先来看 Builder 类。文件路径： ContentBuilder/Builder/Builder.cs。

构建/编译参数

using Microsoft.Xna.Framework.Content.Pipeline;
using MonoGame.Framework.Content.Pipeline.Builder;

// 资源构建参数
var contentCollectionArgs = new ContentBuilderParams()
{
    Mode = ContentBuilderMode.Builder,
    // 所有操作的根目录
    WorkingDirectory = $"{AppContext.BaseDirectory}../../../",
    // 要编译的资源文件的路径
    SourceDirectory = "Assets", 
    // 已编译的资源文件的保存路径
    OutputDirectory = "bin/Content",
    // 临时文件的位置
    IntermediateDirectory = "obj/Content",
    // 编译资源针对的操作系统/平台
    Platform = TargetPlatform.DesktopGL,
    // 日志详细程度
    LogLevel = LogLevel.Info,
    // 是否应对输出文件进行压缩
    CompressContent = false,
    // 是否忽略缓存，强制重新编译资源
    Rebuild = false,
};
var builder = new Builder();

if (args is not null && args.Length > 0)
{
    builder.Run(args);
}
else
{
    builder.Run(contentCollectionArgs);
}

return builder.FailedToBuild > 0 ? -1 : 0;

资源/内容处理规则

在 public class Builder : ContentBuilder 中的 GetContentCollection() 方法内定义资源/内容怎样被处理，这又包括两个过程：

  1. 哪些资源会被处理
  2. 应该怎么处理？

导入资源

Assets 资源目录下的文件夹结构默认情况下会被保留到输出中。自定义输出结构的方法见下文。

使用通配符匹配

WildcardRule 用来定义如何引入、排除特定的文件/文件夹。

通配符模式参考：

模式 匹配
* 任何字符序列
? 任何单个字符
[abc] 括号中的任一字符(多选一）
[!abc] 任一字符，只要不在括号中
{a,b,c} 括号内的任何候选项（参考这个用法*.{png,jpg,bmp}) 
    public override IContentCollection GetContentCollection()
{
    var contentCollection = new ContentCollection();

    // 通配符模式，包含资源文件夹下所有文件
    // 官方推荐保留这一行，在其后编写特定逻辑
    // 移除此行，仅仅包含自定义逻辑也是允许的
    // 所有的 rules 按照顺序求值
    contentCollection.Include<WildcardRule>("*");

    // 引入所有 png 文件
    contentCollection.Include<WildcardRule>("*.png");

    // 引入 Textures 文件夹下所有的文件
    contentCollection.Include<WildcardRule>("Textures/*");

    // 引入任何子目录下的 png 文件
    contentCollection.Include<WildcardRule>("*/*.png");        

    // 指定类型引入文件
    contentCollection.Include<WildcardRule>("*.{png,jpg,bmp}");

    // 复制而不予处理
    contentCollection.IncludeCopy("Data/config.json", "config.json");

    // 引入某个文件，但输出到不同的文件夹
    contentCollection.Include("Sprites/player.png", "Characters/Player.xnb");

    // 排除文件
    // 排除特定文件
    contentCollection.Exclude<WildcardRule>("texture/texture01.png");

    // 排除带有 debug 字样的 png
    contentCollection.Exclude<WildcardRule>("*debug*.png");

    // 排除以 test_ 开头的文件
    contentCollection.Exclude<WildcardRule>("test_*");

    // 排除特定文件夹下所有文件
    contentCollection.Exclude<WildcardRule>("Debug/*");

    // ...
}
使用正则匹配
public override IContentCollection GetContentCollection()
{
    var contentCollection = new ContentCollection();

    // 匹配以 _diffuse.png 结尾的文件
    contentCollection.Include<RegexRule>(@".*_diffuse\.png$");
    // 匹配一个或多个连续数字
    contentCollection.Include<RegexRule>(@"level_\d+\.tmx");
    // 匹配特定文件夹下的特定文件
    contentCollection.Include<RegexRule>(@"^(Textures|Sprites)/.*\.png$");

    // 排除规则里也可以用
    // 后缀 .backup 结尾的备份文件
    contentCollection.Exclude<RegexRule>(@".*\.bak$");
    // 波浪号结尾的临时文件
    contentCollection.Exclude<RegexRule>(@".*~$");
    // ...
}
文件夹管理
public override IContentCollection GetContentCollection()
{
    var contentCollection = new ContentCollection();

    // 图片文件夹
    contentCollection.SetContentRoot("Graphics");
    contentCollection.Include<WildcardRule>("*.{png,jpg,bmp}");

    // 音频文件夹
    contentCollection.SetContentRoot("Audio");
    contentCollection.Include<WildcardRule>("*.{wav,mp3,ogg}");

    // 字体文件夹
    contentCollection.SetContentRoot("Fonts");
    contentCollection.Include<WildcardRule>("*.spritefont");

    // 其他仅需要复制到输出目录的文件
    contentCollection.SetContentRoot("Data");
    contentCollection.IncludeCopy<WildcardRule>("*.{xml,json,txt}");
}

处理资源

资源的完整自定义处理逻辑涉及四个类，下面以自定义处理 MP3 文件为例。

目标，将 MP3 文件读取为 byte[]，并在游戏中通过 ContentManager Content 的 Load<byte[]>("sounds/MP3Name") 载入 MP3 内容。

使用自定义 Importer 和 Processor

先看 ContentBuilder 项目的 Builder.cs 中需要做的工作。

public override IContentCollection GetContentCollection()
{
    var contentCollection = new ContentCollection();

    contentCollection.Include<WildcardRule>("*");

    // 参数 1 指定载入一个 MP3    
    // 参数 2、3 的 Importer 和 Processor 是我们重写的
    // 这里也可以使用 WildCardRule 批量处理 mp3 文件，具体规则在上文有介绍
    contentCollection.Include("sounds/bgm/menuBackground.mp3", new FmodByteImporter(), new FmodByteProcessor());

    return contentCollection;
}
自定义 Importer 和 Processor

在 ContentBuilder 项目中新建两个类：

Importer 负责导入内容:

using Microsoft.Xna.Framework.Content.Pipeline;

namespace ContentBuilder.ContentPipeline;

// 显示名随便写
[ContentImporter(".fmod", DisplayName = "FmodByteImporter", DefaultProcessor = "FmodByteProcessor")]
// 这个泛型参数要写我们的目标类型 byte[]
public class FmodByteImporter : ContentImporter<byte[]>
{
// 这里什么也不做，就是从磁盘上把文件读入 byte[]
    public override byte[] Import(string filename, ContentImporterContext context) => System.IO.File.ReadAllBytes(filename);
}

Processor 负责处理数据:

using Microsoft.Xna.Framework.Content.Pipeline;

namespace ContentBuilder.ContentPipeline;

// 显示名随意
[ContentProcessor(DisplayName = "FmodByteProcessor")]
// 泛型参数同上 byte[]
public  class FmodByteProcessor : ContentProcessor<byte[], byte[]>
{
    public override byte[] Process(byte[] input, ContentProcessorContext context) {
        // 这里接收从 Importer 导入的 byte[]
        // 可以在这里添加解密、裁剪或格式校验逻辑
        // 这里我们直接返回原始数组，不对文件进行解析
        return input;
    }
}
自定义 Writer

在 ContentBuilder 中增加这个类。

Writer 的职责是将数据写出到 .xnb 文件：


using Microsoft.Xna.Framework.Content.Pipeline;
using Microsoft.Xna.Framework.Content.Pipeline.Serialization.Compiler;

namespace ContentPipelineExtension;

[ContentTypeWriter]
public class FmodByteWriter : ContentTypeWriter<byte[]>
{
    protected override void Write(ContentWriter output, byte[] value)
    {
        // 先写长度，再写数据
        output.Write(value.Length);
        output.Write(value);
    }

    // 这一步很关键
    // 涉及游戏主项目能不能正确载入资源
        public override string GetRuntimeReader(TargetPlatform targetPlatform) {
        // 这里的字符串表示的是主项目中对我们载入的 byte[] 资源进行读取的 Reader 类
        // 如果 ContentBuilder 的这个项目引用了这个 Reader 所在的项目
        // 最安全的写法是
        // return typeof(Content.Service.RawAudioReader).AssemblyQualifiedName;
        // 如果没有这样的引用，那么，就参照主项目中实际存在的 Reader 类来写，稍后我们会创建这个类，这里假设它存在
        // 这个字符串由逗号 "," 分割为两部分，第一部分是 Reader 的完全限定名，即包含命名空间在内的类名
        // 第二个部分是你的游戏主项目的程序集名称，一般与你的项目同名
        // 如果实在不知道怎么写，就在 实际的 Reader 类中，写一行下面的语句，然后打印出来看看
        // string correctString = this.GetType().AssemblyQualifiedName; 
        return "BBBall.Service.FmodByteReader, BBBall";
        }

    public override string GetRuntimeType(TargetPlatform targetPlatform) => typeof(byte[]).AssemblyQualifiedName;
}
自定义 Reader

在游戏主项目中创建这个类。

Reader 的任务是运行时从 .xnb 文件中还原数据：

using Microsoft.Xna.Framework.Content;

// 命名空间和类名要和上面 Writer 中的字符串保持一致
namespace BBBall.Service;

public class FmodByteReader : ContentTypeReader<byte[]>
{
    protected override byte[] Read(ContentReader input, byte[] existingInstance)
    {
        // 读取顺序必须与 Writer 一致：先读长度，再读数组
        int size = input.ReadInt32();
                return input.ReadBytes(size);
    }
}
在项目中载入资源

如果之前编译过资源，构建项目时不会二次编译，需要首先到游戏主项目的输出目录下将 Content （资源输出目录）移除，然后在构建项目。构建完成后就可以正常在项目中载入 byte[] 形式存储的 MP3 了。

Audio.Init();
var menuBackground = Content.Load<Byte[]>("sounds/bgm/menuBackground");
if (Audio.GetMusicState() == Audio.PlayState.Stopped)
    Audio.PlayMusic(menuBackground);
